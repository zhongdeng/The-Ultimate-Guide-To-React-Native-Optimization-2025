# 如何分析 React Native 中原生部分的性能

分析不仅对 JavaScript 代码至关重要，我们在[《如何分析 JavaScript 和 React 的性能》](../part1/1.How_to_Profile_JS_and_React_Code.md)一章中介绍过，而且对其他语言和平台（例如 iOS 或 Android）也同样重要。瓶颈会同时出现在JS端和原生端，当 React Native DevTools 不够用时，我们需要转向平台的原生工具——Android Studio 或 Xcode。

除了CPU、内存和网络性能分析外，电池使用情况对移动用户来说同样重要。如果不考虑屏幕亮度，电池使用情况通常可以作为CPU开销的一个指标。

Xcode提供了“调试导航器”视图，可以一目了然地检查CPU、内存、磁盘和网络负载。当你的应用程序运行并连接到Xcode调试器时，它位于Xcode侧边栏的“调试导航器”图标下：

CPU监视器测量一段时间内完成的工作量。百分比值显示相对于单个核心的使用情况，因此在React Native应用中，它可能会超过100%。内存监视器观察应用程序的内存使用情况。所有iOS设备都使用SSD进行永久存储——访问这些数据比访问RAM要慢。磁盘监视器了解应用程序的磁盘读写性能。网络监视器分析iOS应用程序的TCP/IP和UDP/IP连接。

你可以点击每个监视器以获取更多信息。Xcode还提供了一个默认情况下不显示的额外监视器，但它可以帮助你检查UI——这就是视图层次结构，我们在“使用视图扁平化”一章中讨论过。

Xcode 预装了一个名为 Instruments 的调试和性能分析工具。它是一套检查工具，每个工具都有不同的用途。您可以从模板列表中进行选择，并根据您的目标选择其中任何一个：提升性能、延长电池寿命或修复内存问题。对于 CPU 性能分析，我们将使用 Time Profiler。让我们深入研究一下。

打开 Xcode 后，从 Xcode > 打开开发者工具 > Instruments 菜单中选择 Instruments。

向下滚动找到 Time Profiler 工具，其中包含 CPU Profiler。

它将打开一个新窗口。要开始分析您的应用，请点击目标设备（此处为 iPhone 16 Pro）旁边的下拉菜单，选择应用 (SampleApp)，然后点击录制按钮（红色圆圈）：

分析器将重新启动您的应用，并从头开始收集 CPU 样本进行分析。现在，您可以以某种方式使用您的应用，以重现您或您的用户发现并想要调查的性能回归问题。获得足够的数据后，在 Instruments 窗口中停止录制。

在我们的例子中，我们正在渲染一个在 ScrollView 中呈现 5,000 个视图的应用，并且每次按下按钮状态更新时都会重新渲染。除此之外，我们还将切换一个图像，该图像会将列表向下推，从而导致重新布局。

在我们的例子中，我们关注的是按下按钮并等待应用更新时，大约占总时间的 1.3 秒。您可以注意到 UI 线程（标记为 SampleApp）上出现了一个短暂的峰值，随后 JavaScript 线程上出现了一些密集的 CPU 工作，UI 线程上也有大量工作。配置文件在底部显示为火焰图，显示了 React Native 的内部调用点（我们过滤掉了系统库）。

请注意，在 UI 线程 CPU 负载较高时，会出现“微挂起”，而 JS 线程负载较高时则不会出现。这表明 UI 线程正在执行大量工作，因此无法立即与我们的内容进行交互。不过，情况可能会更糟——如果线程完全阻塞，它会显示为“挂起”，您应该优先处理。挂起工具与时间分析器 (Time Profiler) 配合使用，可以快速发现应用流程中存在问题的部分。

您可以有效地阻塞 JS 线程，用户仍然可以与应用的原生 UI 元素交互，例如按下按钮。它只是不会立即与 JS 线程进行通信。这是 React Native 一个很棒的设计理念，我们经常将其视为理所当然，值得赞赏。

CPU 分析器工具允许进行大量自定义，以帮助您查明问题的根源。例如，您可以过滤调用树以隐藏系统库，或者完全反转调用树以访问您可能从其他分析器中了解到的自下而上的视图，并按“计数”表示的权重进行排序（例如，Mc 表示“百万计数”，表示收集了数百万个样本）。它本质上允许您查看耗时最多的函数，无论调用它们的是哪个高级函数。如果有帮助，您还可以按线程检查调用情况。

您可以在调用树和火焰图视图之间切换，您可能已经从 React Native DevTools 中了解过火焰图。在我们的初始屏幕截图中，我们固定了 JavaScript 线程，以查看它与 UI 线程的关联以及检测到的挂起。

借助 CPU 分析器，我们可以得出结论，在 ScrollView 中存储 5k 个元素对于我们应用的性能而言并非最佳，我们需要采取一些措施。

对于 Android 平台，我们可以使用 Android Studio 和内置工具来分析应用的 CPU、内存、网络和电池使用情况。

Android Studio 是由 JetBrains 开发的 IDE。它由 Google 官方支持，是官方的 Android IDE，可用于开发任何 Android 应用。它功能强大，包含许多功能，包括 Android Profiler。

要打开 Profiler，请在 Android Studio 菜单栏中选择“视图”>“工具窗口”>“Profiler”：

或者在 Android Studio 工具栏中点击“Profiler: Run 'app' as profileable”。如果此操作无效，您需要运行可调试的版本，或者根据官方 Android 开发者指南自行创建并运行可分析的应用版本。

现在，您应该能够连接到设备上正在运行的可分析应用。我们将使用“查找 CPU 热点”任务，然后点击“启动分析器任务”按钮，开始从我们的应用中收集样本。

值得注意的是，Android 设备的生态系统比 iOS 更加分散。手机、平板电脑、电视等设备种类繁多，从低端到中端再到高端。请选择可用的最低端设备或模拟器进行分析。如果可能，请使用实时用户监控数据。

我们将执行与 iOS 应用相同的场景：按下按钮几次，即可重新渲染一个庞大的列表。停止分析后，UI 将变为火焰图，显示 CPU 和各个线程的工作细目。

让我们放大一下按下按钮后 JS 线程的情况；我们可以看到 CPU 使用率有所上升：

我们可以观察到 Hermes JS 引擎中出现了许多微小的 1ms 峰值活动，这些峰值处理着最终需要在屏幕上绘制的所有 5k 个视图。所有这些工作总计超过 240 ms，远远超过了实现至少 60 FPS 所需的 16.6 ms 的预算。请注意，大约三分之一的总时间都花在了 React 协调算法的“提交”阶段，该阶段使用 Yoga 布局新的视图，然后将其挂载并绘制到屏幕上。

如果我们在火焰图上稍微向上滚动查看主线程，我们会注意到按下按钮时有一个事件处理程序，随后松开手指时又有一个事件处理程序。

触摸释放的时间几乎与 JS 线程上增加的工作量的开始时间完全吻合，这是通过 React Native 内部的同步 JSI 调用实现的。在传统架构中，这些信息会被桥接器序列化并广播，从而增加此交互的额外开销。

与其他任何性能分析工具一样，我们也可以访问自上而下或自下而上的调用树分析，并根据我们感兴趣的库和调用进行筛选。

值得一提的是，Android Studio 或 Chrome JS Profiler 等性能分析工具允许您导出跟踪记录并将其加载到第三方工具中。

Perfetto 就是这样一款可能派上用场的工具，它允许您通过加载之前导出的跟踪记录来进行系统性能分析、应用跟踪和跟踪分析。它为 Android Studio Trace Viewer 提供了额外的功能。您可以通过 ui.perfetto.dev 上的在线查看器上传跟踪记录，并从不同的角度进行进一步的检查。

了解如何使用性能分析器是您可以学习的最重要技能之一，它可以帮助您修复任何代码库中的任何性能问题。它对于您尚未编写的代码尤其有用。但是使用正确的工具，您可以了解其性能特征并在需要时应用优化。
