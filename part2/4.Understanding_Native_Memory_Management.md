# 理解 Turbo Modules 和 Fabric 的线程模型

在原生编程语言中，开发者在应用程序运行期间通常或多或少需要负责内存和资源的管理。这种管理方式取决于语言和平台。内存管理大致可以分为两大类：手动管理和自动管理。

当使用手动内存管理时，例如在 C 或 C++ 这样的语言中，我们需要负责分配和释放内存（以及其他资源）。这种方式通常冗长且容易出错；不过，如果使用得当，它能提供最佳的性能。

在大多数情况下，你应该选择自动内存管理来使代码更易理解和维护。然而，如果你希望从代码中榨取出每一纳秒的性能，并且你对如何管理内存有深刻理解，那么手动管理可能更适合你。举例来说，这种做法在游戏开发中就很常见。

了解编程语言中常用的内存管理模式至关重要，因为这能帮助你更好地理解自己编写的代码，从而提升代码的效率和可预测性。我们从探索自动内存管理开始。

## 内存管理模式

在这种方式中，内存是通过垃圾回收算法自动管理的。我们可以强调两种常见的方法，各有利弊。

### 引用计数

在移动开发中，这种模式被 Objective-C 和 Swift 使用（在 Swift 中称为自动引用计数，ARC），但它也存在于 Python、PHP 等语言中。

每当我们传递一个对象的引用时，编程语言的引用计数器就会增加其使用次数。这包括将对象传递给另一个对象，或作为函数参数传递。当引用被终止（例如超出作用域）时，使用计数会减少。如果使用计数降为零，该对象就会被释放，其资源也随之释放。

尽管它可能会影响运行时性能，但它的优点是能在最后一个引用超出作用域时立即清除资源，释放内存。虽然这听起来像是一个理想的方案，但我们必须注意“引用循环”（Reference Cycles），当两个对象互相持有“强引用”时，就会产生引用循环。这会将引用计数锁定在 1，从而阻止对象被释放。解决这个问题的常见方法是使用弱引用（**weak** reference），尽管这也可能引入引用循环的风险。

### 垃圾回收器（Garbage Collector）

这种内存管理方式被 JavaScript、Java、C# 等语言采用。

垃圾回收器（GC）的工作独立于你的代码运行逻辑。你可以将其想象为一个独立的进程，例如 Hermes 引擎会启动一个单独的线程来运行其 Hades GC。GC 可以定期运行，也可以在对象分配时触发。垃圾回收器会扫描程序内存，查找不再可访问的对象。然后这些不可访问的对象会被释放。

这种方式不会遇到引用循环的问题。但垃圾回收器启动的时机是不可预测的，我们无法控制对象何时被销毁。此外，如果 GC 算法不是并发执行的，那么应用在 GC 运行期间会被暂停，这可能会影响用户的使用体验。

### 手动管理

底层语言，比如 C 或 C++，允许我们完全控制内存的分配和释放。通常，你可以在两个地方分配内存：栈（stack）和堆（heap）。

栈是一个按后进先出（LIFO）方式存储数据的内存区域，内存在其中是自动分配和释放的。它非常快，但大小和作用域有限。而堆是用于动态分配的更大内存区域，其中的数据可以在全局访问。当你在堆上分配内存（例如使用 new 或 malloc）时，你需要在不再需要时手动释放它。

这种技术在使用得当时最为高效；但如果你不了解其中细节，很容易“搬起石头砸自己的脚”。

## 管理内存

现在你已经了解了理论知识，接下来我们将进入实践环节！本节将介绍 Kotlin、Swift 和 C++ 中的内存管理方式，让你更深入地理解如何在这些语言中正确地管理内存。

### C++

在 C++ 语言中，内存管理是显式且手动的。然而，过去那种必须通过 `new` 和 `delete` 操作所有内存的时代早已过去。现在你可以使用智能指针（smart pointer），它由 C++ 标准库（std）提供。通过这种方式，你无需手动分配和释放内存，从而在应用中减少内存泄漏的可能性。

> 以下内容假设你已对 C 和 C++ 中的指针概念有所了解。如果不熟悉，你可以先学习该概念，或跳过此部分。

你可以使用多种智能指针：

- **std::unique_ptr\<T>** —— 不能被复制到另一个 `unique_ptr`，不能按值传递给函数，也不能用于要求拷贝的标准库算法。`unique_ptr` 只能被移动。

- **std::shared_ptr\<T>** —— 引用计数的智能指针。当你希望将一个原始指针分配给多个所有者时使用，例如从容器返回指针副本但又希望保留原始指针的情况。

- **std::weak_ptr\<T>** —— 与 `shared_ptr` 配合使用的特殊智能指针。它访问的是由一个或多个 `shared_ptr` 拥有的对象，但不参与引用计数。

最常用的智能指针是 `std::shared_ptr<T>`，它是一个为指定类型提供引用计数机制的封装对象，这种机制与 Obj-C 和 Swift 中的方式类似。我们来看一下智能指针的使用示例：

首先是 `std::unique_ptr`：

```C++
void takeOwnership(std::unique_ptr<std::string> s1) {
  std::cout << *s1;
  // Gets automatically deleted when function ends
}

int main()
{
  auto str1 = std::make_unique<std::string>("Hello World");

  std::cout << *str1;

  // Can only be moved. Doesn't allow for copying
  takeOwnership(std::move(str1));

  // str1 is cleared here
  return 0;
}
```

在上面的例子中，我们使用 `make_unique` 创建了一个新的唯一指针，该指针持有一个 `std::string`。然后我们使用解引用操作符打印字符串的值。该例还展示了该智能指针的唯一性，它只能通过移动（move）来转移所有权。这在构建不可变数据结构时很有用。

接着我们看看 `std::shared_ptr`，我们继续用相同的示例便于理解：

```C++
void takeOwnership(std::shared_ptr<std::string> s1) {
  std::cout << *s1;
  // After function returns, reference count gets back to 1
}

// We accept a reference
void takeReference(const std::shared_ptr<std::string> &s1) {
  std::cout << *s1;
}

int main()
{
  // Create a shared pointer
  auto str1 = std::make_shared<std::string>("Hello World");
  std::cout << *str1;

  // We increase the reference count by 1 and create a copy of the pointer (not underlying values)
  takeOwnership(str1);

  // Since we didn't move the object it can be still accessed here
  std::cout << *str1;

  // We don't alter the reference count since we pass a reference here
  // This doesn't copy anything
  takeReference(str1);

  std::cout << *str1;

  return 0;
}
```

在上例中，我们使用 `std::make_shared()` 创建了一个新的共享指针，然后用解引用操作符将其打印出来。我们将它传递给 `takeOwnership` 函数，该函数通过值传递（拷贝指针而非底层值），这会增加引用计数。

由于共享指针可以被拷贝，值不会被清除，因此我们仍然可以使用它。随后我们传递给 `takeReference` 函数，而这个函数只是引用传递，因此不会增加引用计数。当函数的使用不会超出调用者的生命周期时，应使用引用传递；如果希望 `shared_ptr` 独立于调用者存活，应使用值传递。

现在我们来看 `std::weak_ptr` 的使用方法：

```C++
void useWeakPtr(std::weak_ptr<std::string> weak) {
  // Try to get access to the object
  // .lock() converts weak to shared_ptr if object exists
  if (auto shared = weak.lock()) {
    std::cout << *shared;         // Safe to use
    // shared_ptr reference count temporarily increases here
  } else {
    std::cout << "Object no longer exists!\n";
  }
} // shared goes out of scope, count decreases

int main() {
  // Create a shared pointer
  auto str1 = std::make_shared<std::string>("Hello World");

  // Create a weak pointer from shared pointer
  std::weak_ptr<std::string> weak1 = str1;  // Doesn't increase reference count

  std::cout << "Reference count: " << str1.use_count() << "\n"; // Shows 1

  // Use the weak pointer
  useWeakPtr(weak1);  // Object exists

  // Reset the shared pointer
  str1.reset();  // Reference count becomes 0, object is destroyed

  // Try to use weak pointer again
  useWeakPtr(weak1);  // Will print "Object no longer exists!"

  return 0;
}
```

我们可以从 `shared_ptr` 获取一个 `weak_ptr`，它不会增加引用计数。要使用 `weak_ptr`，需要调用 `.lock()` 方法将其转换为 shared_ptr，前提是该资源仍然存在。

### Android

在 Android 平台上，我们主要使用 Kotlin，其运行在 JVM（Java 虚拟机）上，使用垃圾回收机制来管理内存。与 JavaScript（同样使用 GC）类似，如果不释放资源，也可能引发内存泄漏。例如，忘记取消注册监听器或中止长时间运行的后台操作。

我们能做的少数几件事之一，就是使用弱类型容器，比如 `WeakHashMap`（JavaScript 中有类似的 `WeakMap`）。这类容器不会对其项目创建强引用。因此，如果没有其他引用指向它们，GC 会在下一次执行中将其回收。下面是 `WeakHashMap` 的使用示例：

```Kotlin
fun main() {
  val weakMap = WeakHashMap<String, String>()

  // Add entries inside a scope
  run {
    weakMap[String("temp key")] = "some value"
    println("Map size: ${weakMap.size}") // Prints: 1
    println("Value: ${weakMap[key]}") // Prints: some value
  }

  // Force GC for demonstration
  System.gc()
  Thread.sleep(100)

  println("Map size after GC: ${weakMap.size}") // Prints: 0
}
```

需要注意的是，`WeakHashMap` 仅对其键使用弱引用。如果你需要对存储在映射中的值使用弱引用，可以使用 `WeakReference()`，如下所示：

```Kotlin
weakMap["key"] = WeakReference(LargeObject("1"))
```

### iOS

如前所述，Swift 和 Objective-C 在整个生态系统中都使用引用计数机制。我们来通过一个简单的示例了解跨作用域时引用计数如何变化。我们将使用一个 Person 类做演示：

```Swift
class Person {
  let name: String
  init(name: String) {
      self.name = name
  }
}

do {
  let person1 = Person(name: "John") // Reference Count: 1

  do {
    let person2 = person1  // Reference count: 2
  } // person2 goes out of scope, Reference count: 1

} // person1 goes out of scope, Reference count: 0
```

当我们将 `person1` 赋值给 `person2` 时，引用计数会增加（我们没有创建副本！）。然后，在 `do` 块作用域结束后，引用计数回到 1，最后降为 0。

## 内存泄漏的来源

下面我们来检查几个导致应用内存泄漏的常见模式。

### 忘记在手动管理中释放资源

在使用手动内存管理时，极易引发内存泄漏。如前所述，选择手动管理就意味着要自行负责。能力越大，责任越大。我们来看一个例子说明引发内存泄漏有多简单：

```C++
int main()
{
  std::string *str1 = new std::string {"Hey"};

  std::cout << *str1;

  // Oops! Memory leak, we forgot to delete str1.

  return 0;
}
```

你分配了一个新的 `std::string`，打印其值，但最后忘记释放该字符串，因此引发了内存泄漏。解决方法是在资源不再需要时调用 `delete`：

```C++
int main()
{
  std::string *str1 = new std::string {"Hey"};
  // ...
  delete str1;

  return 0;
}
```

另一种做法是将其分配在栈上（去掉 **new**），这样在其超出作用域时会自动释放：

```C++
int main()
{
  std::string str1 = std::string {"Hey"};

  cout << str1;

  return 0; }
```

在将某样东西放到堆上之前你应三思，因为堆的开销远大于栈。同时，也要谨慎在栈上分配太多，否则可能导致……栈溢出（stack overflow）！但如果可能，优先使用栈，只有在必要时才使用堆。

### 引用循环

在使用引用计数的系统中，最常见的内存泄漏来源是引用循环，也叫循环引用。我们来看一个简单的示例：

```C++
class A {
  std::shared_ptr<B> b;
};

class B {
  std::shared_ptr<A> a;
};
```

```Swift
class A {
  var b: B?
}

class B {
  var a: A?
}
```

在这个例子中，类 A 依赖于类 B，而类 B 又依赖于类 A，从而造成了引用循环。

> 值得注意的是，在真实项目中，引用循环通常更加复杂，可能跨越多个类，因此更难发现和修复。

解决方法是通过引入弱指针 `(std::weak_ptr<T>)` 来打破循环。弱指针是一种可选依赖，它不会阻止被依赖对象被销毁。如果我们想访问该依赖，需要先检查它是否仍然存在，因为此时已不再保证：

```C++
class A {
  std::shared_ptr<B> b;
};

class B {
  std::weak_ptr<A> a;
};
```

```Swift
class A {
  var b: B?
}

class B {
  weak var a: A?
}
```

另一种解决方法是引入第三个类 **C**，持有类 **A** 和 **B** 依赖的共享资源，从而避免引用循环。这种重构技术在 JavaScript 项目中也非常常见。

### 未释放的资源

在使用垃圾回收机制的语言中，如果不释放资源，也可能引发内存泄漏。例如使用监听器模式却忘记移除监听器。如果你有 JavaScript 背景，这类问题你可能早已熟悉。我们来看一个展示该问题的示例。首先，创建一个简单的 `DataManager` 类，负责注册并调用监听器：

```Swift
// Simple data manager class
interface DataListener {
  fun onDataChanged(data: String)
}

class DataManager {
  // List to store all registered listeners
  private val listeners = mutableListOf<DataListener>()

  // Method to register listeners
  fun registerListener(listener: DataListener) {
    listeners.add(listener)
  }

  // Method to unregister listeners
  fun unregisterListener(listener: DataListener) {
    listeners.remove(listener)
  }

  // Method to notify listeners of data changes
  fun notifyDataChanged(data: String) {
    listeners.forEach { it.onDataChanged(data) }
  }
}
```

当我们使用该类时，必须记得取消注册监听器，否则就会引发内存泄漏：

```Swift
class MyClass {
  private val dataManager = DataManager()

  private val listener = object : DataListener {
    override fun onDataChanged(data: String) {
      println("Data changed: $data")
    }
  }

  init {
    // Register listener but never unregister
    dataManager.registerListener(listener)
  }

  // Memory leak: No way to unregister the listener
  // MyClass instance will never be garbage collected
  // as DataManager holds a strong reference to the listener
}
```

由于 Kotlin 没有内建的确定性析构器或反初始化器（deinitializer），我们需要使用如 `AutoCloseable` 的生命周期机制：

```Kotlin
class MyClass : AutoCloseable {
  override fun close() {
    // Cleanup code here
    dataManager.unregisterListener(listener)
  }
}
```

我们也可以在 `DataManager` 中使用 `WeakReference` 存储回调，这样会自动处理这类问题。

```Kotlin
class DataManager {
  // List to store all registered listeners
  private val listeners = mutableListOf<WeakReference<DataListener>>()

  // ...
}
```

# 使用手动重载

即使是提供自动内存管理的语言，也通常提供手动干预机制，主要用于与外部语言的互操作性。虽然这种做法很有用，但如果使用不当，也可能成为内存泄漏的来源，且难以发现。

例如在 Swift 中，我们可以通过 `Unmanaged` 类型手动增加或减少引用计数。使用时必须格外小心，否则可能导致崩溃或非预期行为。`Unmanaged` 类提供了一些方法来手动管理 ARC 行为：

- **passRetained** —— 创建一个 **Unmanaged** 实例并增加引用计数。
- **passUnretained** —— 创建一个不增加引用计数的 **Unmanaged** 实例。
- **takeRetainedValue** —— 获取对象并减少引用计数。
- **takeUnretainedValue** —— 获取对象但不影响引用计数。

我们来看一个在 Swift 类中使用 `Unmanaged` 的示例（通常用于与 C 交互）：

```Swift
class MyObject {
  deinit { print("Deallocated") }
}

let obj = MyObject()                        // Reference Count = 1
let unmanaged = Unmanaged.passRetained(obj) // Reference Count = 2
let object1 = unmanaged.takeRetainedValue() // Reference Count = 1
```

然而，如果你将引用计数减少到小于 0，就会导致崩溃！

```Swift
let obj = MyObject()                          // Reference Count = 1
let unmanaged = Unmanaged.passUnretained(obj) // Reference Count = 1(no change)
let object1 = unmanaged.takeRetainedValue()   // CRASH!
```

一条经验法则是：**passRetained** 必须配对 **takeRetainedValue** **使用，passUnretained** 必须配对 **takeUnretainedValue** 使用。你还可以在 **unmanaged** 对象上调用 `toOpaque()` 方法来获取原始指针，以便传递给 C/C++。

```Swift
unmanagedObject.toOpaque() // 0x000056076df5b1a0
```
