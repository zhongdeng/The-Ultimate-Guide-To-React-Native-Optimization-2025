# 理解不同平台的差异

作为一名 JavaScript 开发者，你很可能已经熟悉一些流行的 IDE 或编辑器，并且在进行项目开发时会坚持使用自己喜欢的那一个。

然而，当你开始使用 React Native 进行开发时，可能会因为在设置指南中出现的大量新 IDE 和工具而感到不知所措。这既是好事，也是 React Native 作为原生框架带来的副作用。正因如此，你时不时会需要同时使用 Android Studio 和 Xcode。

熟悉这些工具将帮助你更好地阅读本指南，并提升你在 React Native 中的开发效率。在本指南中，我们将重点介绍三个平台之间的共性、关键差异，以及提升工作流效率的技巧。

这些信息将帮助你理解本指南的其他章节，尤其是在我们提到 Android Studio、Xcode 或不同构建阶段时。

> 如果你使用 [Expo](https://expo.dev/) 来构建应用程序，你可能不需要太频繁地离开自己熟悉的 IDE。Expo 通过使用 Dev Clients、Config Plugins 等很好地抽象了原生部分。本指南的内容主要针对依赖于 React Native Community CLI 的项目。

## 前提条件

在继续之前，请确保你已经准备好了 Android Studio 和/或 Xcode 开发环境。如果你已经按照 [React Native 官网](https://reactnative.dev/docs/next/environment-setup)的设置指南操作过，那么这些工具应该已经安装在你的机器上了。如果你还没有完成这一步，现在是一个很好的时间去做。React Native 的文档提供了详细的解释，因此我们将在本指南中跳过这一主题。

> 虽然 Web、iOS、Android 各个平台各有不同，但它们之间也有一些相似之处。认识到这一点将使你作为 React Native 开发者的工作变得稍微轻松一些。

## IDEs

首先也是最重要的一点，在构建应用时，你至少需要一个编辑器来编写和编辑代码。对于 JavaScript，你通常会使用自己最熟悉的编辑器——例如 VSCode、WebStorm 或 Neovim 等。由于这些工具本身对工作流限制较少，因此最佳实践是将你的开发环境配置为支持 TypeScript、ESLint、Prettier 和 React。

对于 Android 和 iOS，一般推荐使用分别由 Google 和 Apple 提供的官方 IDE。虽然你可以在 VSCode 中编写 Swift 或 Kotlin，但上述官方 IDE 能够提供更为可靠和可预测的开发环境。

这些原生 IDE 自带的一些功能包括：

- **View Hierarchy Debugger** —— 可视化地检查和调试 UI 布局，允许你分析元素定位、约束关系以及渲染问题。

- **Instrumentation & Performance Profiling** —— 内置工具用于监控 CPU、内存、电池使用情况以及 UI 响应性，例如 Android Studio 的 Profiler 和 Xcode 的 Instruments。

- **Advanced Debugging Tools** —— Xcode 支持 LLDB 和 GDB，Android Studio 提供 Logcat 和 Debugger，能深入分析堆栈跟踪、崩溃报告以及内存泄漏。

- **Seamless Gradle (Android) & Build System (Xcode) Integration** —— 可以无需额外配置管理依赖项、构建变体、签名和打包；并集成了代码签名、应用验证及直接发布到应用商店的工作流。

- **Automated Testing Tools** —— 支持本地单元测试和 UI 测试框架，如 XCTest（Xcode）和 Espresso/JUnit（Android Studio），且内置了测试运行器。

在接下来的指南和最佳实践部分，我们将依赖这些工具。

## 依赖管理

在开发 Web 或移动应用时，我们通常会依赖第三方模块。在 React Native 中，一个库可能同时包含 JavaScript、iOS 和 Android 的依赖。了解各平台的依赖管理方式，有助于你在需要直接处理原生模块时更为得心应手。

React Native 通过 **Autolinking** 机制抽象了大部分复杂性，它能自动检测并链接专为 React Native 开发的模块。然而，Autolinking 只适用于专门为 React Native 构建的库。如果一个依赖项包含了 iOS 和 Android 的原生代码，但并不是以 React Native 模块的结构组织的，那么就需要额外的手动步骤。

接下来，我们将分别看看 JavaScript、iOS 和 Android 的包管理方式。理解这个过程，将有助于你管理项目依赖并在需要时引入额外的原生模块。

### JavaScript

对于 JavaScript 依赖，React Native 主要依赖 npm 注册表（npm registry），这是一个集中存放 JavaScript 库的仓库。这个仓库中的包可能仅包含 JavaScript 代码，也可能包含 JavaScript 和原生代码的组合。

在 React Native 项目中安装一个 JavaScript 依赖非常简单，只需使用 npm：

```bash
npm install react-native-bottom-tabs
```

这个命令会从 npm 注册表拉取对应包，添加到 node_modules/ 目录，并更新 package.json 文件。

以下是一些需要注意的文件：

- **package.json** —— 定义项目依赖、脚本和元数据。

- **node_modules** —— 本地存储所有安装的依赖。

- **package-lock.json** —— 确保各环境之间版本的一致性。

> JavaScript 开发者通常还会使用其他包管理工具，比如 Yarn、PNPM 或 Bun。

### iOS

对于 iOS，React Native 使用 CocoaPods 来管理原生依赖。

不同于传统 iOS 项目直接通过 CocoaPods 仓库（Podspec 仓库）或 Swift Package Manager 获取依赖，React Native 库通常将其原生代码发布到 npm 注册表，CocoaPods 通过本地引用的方式来定位和加载这些代码。但如果你需要添加一个不是 React Native 库的原生 iOS 依赖，那么仍需直接通过 CocoaPods 获取。

例如，要将 SDWebImage 添加到你的项目中，你需要在 Podfile 中添加以下内容：

```ruby
pod 'SDWebImage', '~> 5.0'
```

如果你是通过 React Native Community CLI 创建的新项目，最终的结果可能类似于：

```ruby
require_relative '../node_modules/react-native/scripts/react_
    native_pods'
require_relative '../node_modules/@react-native-community/cli-
    platform-ios/native_modules'

platform :ios, '12.4'
install! 'cocoapods', :deterministic_uuids => false
target 'HelloWorld' do
  # Here goes your native dependency
  pod 'SDWebImage', '~> 5.0'
  # Rest of Podfile
end
```

> 在 CocoaPods 中，~> 运算符遵循一种类似于 SemVer（语义化版本控制）的悲观版本约束规则，但略有不同：
>
> - ~> 5.0 允许使用版本 >= 5.0 且 < 6.0（允许补丁和次版本更新，但不允许大版本更新）。
>
> - ~> 5.0.1 允许使用版本 >= 5.0.1 且 < 5.1（仅允许补丁更新，不允许次版本更新）。

添加好依赖后，进入 ios 目录并使用 pod install 命令安装 pods：

```bash
cd ios && bundle exec pod install
```

注意，我们并不是直接运行 CocoaPods，而是使用 bundle 命令。由于 CocoaPods 是一个 Ruby 项目，我们可以借助 Ruby 的依赖管理工具 Bundler，以便在每个 React Native 项目中使用 Gemfile 指定的最合适的 CocoaPods 版本。

> 安装 pods 后，你可以通过 xed . 命令快速启动 Xcode。

以下是一些需要注意的文件：

- **Podfile**——定义 iOS 应用所需的依赖项。

- **Gemfile**——定义给定 React Native 应用所需的 CocoaPods 版本及其他插件。

- **Pods**——存放所有通过 CocoaPods 安装的依赖，该文件夹在安装依赖时生成，通常不应提交到版本控制（默认已在 .gitignore 中列出）。

- **.xcworkspace**——由 CocoaPods 生成的工作区文件，打开 Xcode 项目时必须使用它而非 .xcodeproj，以确保所有 CocoaPods 管理的依赖被正确链接。

> 虽然 Apple 的 Swift Package Manager（SPM）也是一种原生 iOS 项目的依赖管理方式，但 React Native 尚未支持，因此 CocoaPods 仍是默认选择。

### Android

对于 Android，React Native 使用 **Gradle**（Android 的构建自动化工具）来管理依赖。与 iOS 类似，React Native 特定模块并不是通过像 Maven Central 或 JCenter 这样的外部仓库获取的，而是直接从 **node_modules/** 本地引用并编译到项目中。

如果你需要添加一个不是 React Native 模块的原生 Android 依赖（例如用于图片加载的 Glide），则必须手动将其添加到 **android/app/build.gradle** 文件中:

```gradle
dependencies {
  implementation 'com.github.bumptech.glide:glide:4.12.0'
}
```

在 **build.gradle** 中添加新依赖后，需要同步 Gradle 项目以拉取并应用更改

虽然 Gradle 不使用 ~>，但提供了其他灵活的版本管理选项：

- **精确版本**——例如 **com.github.bumptech.glide:glide:4.12.0**

- **版本范围**（+ 运算符）——例如 **com.github.bumptech.glide:glide:4.+**. 允许使用 4.x.x 系列中的最新小版本或补丁版本，但避免大版本升级。

- **动态版本**——例如 **com.github.bumptech.glide:glide:+**. 允许使用最新版本，但通常不推荐，因为这可能会在不经意间破坏你的代码。

- 对于生产应用，**始终使用精确版本**，以确保兼容性，就像 CocoaPods 的 ~> 在 iOS 项目中防止破坏性更新一样。

React Native 项目自带 Gradle Wrapper（gradlew），确保所有开发者使用相同的 Gradle 版本，以避免兼容性问题。可以通过如下命令运行：

```bash
cd android && ./gradlew clean
```

完成后，你就可以在项目中访问新的原生依赖了。

另外，还有一些需要注意的文件：

**build.gradle（项目级）**——位于 android/build.gradle，用于配置全局项目设置，如 Gradle 插件版本、依赖仓库（如 Maven Central 或 JitPack）等。

**build.gradle（应用级）**——位于 android/app/build.gradle，用于定义应用特定的依赖项、构建配置和编译选项。

**gradle.properties**——包含优化 Gradle 构建的配置标志和设置（如启用 Hermes 或 ProGuard）。

**gradlew & gradlew.bat**——React Native 自带的 Gradle Wrapper 脚本，确保所有开发环境中的 Gradle 版本一致，其中 .bat 文件用于 Windows 系统。

**.gradle/** 目录——存储 Gradle 的缓存、编译脚本和临时构建文件。该目录在构建过程中生成，通常不应提交到版本控制。

> 例如，可以通过运行 **./gradlew tasks** 来列出可用的 Gradle 任务。

## 构建项目

在使用 React Native 开发时，理解 JavaScript、iOS 和 Android 如何处理代码执行与编译是非常重要的。与使用编译型语言的原生平台不同，JavaScript 没有传统意义上的构建系统，而是经过不同的转换过程。

### JavaScript

JavaScript 是一种解释型语言，不需要像 Swift 或 Kotlin 那样的单独编译步骤。
不过，由于 ECMAScript 版本众多且 JavaScript 标准不断演变，React Native 需要通过一种称为转译（transpilation）的过程来确保在不同环境中的兼容性。

在 React Native 中，这一过程由 Metro 负责，它是一个 JavaScript 打包器，会将现代 JavaScript 代码转换为 JavaScript 引擎（例如 Hermes）可以理解的版本。在底层，Metro 依赖 Babel，它会重写 JavaScript 语法，以匹配 React Native 中使用的 JavaScript 引擎所支持的最新版本。

> 由于 React Native 依赖的 JavaScript 引擎会随着时间推移而演变，所支持的具体 JavaScript 版本也取决于 Hermes 的版本。React Native 的 Babel 预设（preset）确保 JavaScript 始终被转译为与当前 React Native 中使用的引擎兼容的版本。

### iOS

与 JavaScript 不同，Swift 和 Objective-C 是编译型语言。
iOS 的构建系统会在执行前将源代码编译成机器码，从而在苹果设备上实现优化的性能。

React Native 在 iOS 上依赖 Xcode 的构建系统，后者使用 Clang（用于 Objective-C）和 LLVM（用于 Swift）将人类可读的源代码转换为优化后的二进制文件（如 **.app** 包）。

以下是关键步骤：

- **源代码编译**：使用 Clang/LLVM 将 Swift 和 Objective-C 文件编译为机器码。

- **链接**：将所有已编译的代码、系统框架以及通过 CocoaPods 引入的外部库链接在一起。

- **签名**：苹果要求应用在设备上运行前必须使用有效的证书和配置描述文件进行签名。

- **打包**：生成最终的二进制文件（**.ipa** 文件），其中包含应用及其资源。

### Android

与 iOS 类似，Android 也使用编译型构建系统，但它依赖 Gradle 来自动化构建、测试和打包 Android 应用的过程。

React Native 在 Android 上会将 Java/Kotlin 源代码编译为 DEX（Dalvik Executable）格式，以便在 Android Runtime (ART) 中高效执行。

以下是关键步骤：

- **源代码编译**：使用 Java/Kotlin 编译器将 Java/Kotlin 文件编译为 .class 字节码文件。

- **DEX 转换**：将已编译的 .class 文件转换为 .dex 文件，以便在 Android Runtime 中优化执行。

- **资源处理**：打包 XML 布局文件、图片及其他资源。

- **链接**：链接通过 Gradle 管理的外部库和依赖项。

- **签名**：Android 要求应用使用有效的 keystore 文件进行签名。

- **打包**：生成最终的 .apk 或 .aab 文件，以供安装到设备上。

## 在设备上运行应用

测试是移动应用开发流程中必不可少的一部分。与可以直接在浏览器中测试的 Web 应用不同，移动应用需要通过模拟器（Android 中称为模拟器 emulator）或真实设备来进行测试。模拟器允许开发者在一个受控环境中运行和调试应用，模拟真实设备的行为。

### Android

你可以通过以下方法与 Android 模拟器交互：

- Android Studio AVD Manager：内置的 Android 虚拟设备管理器，可用于创建、配置和启动虚拟设备。

- 命令行（**adb** 和 **emulator** 命令）：

  - **emulator -list-avds** —— 列出所有可用的虚拟设备。

  - **emulator @Pixel_6_API_34** —— 启动特定的虚拟设备。

  - **adb devices** —— 列出正在运行的模拟器和连接的真实设备。

> CLI 工具（如 Expo CLI 或 React Native Community CLI）在底层使用这些命令，以提供更好的开发体验。

### iOS

你可以通过以下方法与 iOS 模拟器交互：

- Xcode：苹果官方的工具，用于在模拟设备上运行 iOS 应用。可以通过 Xcode > Open Developer Tool > Simulator 启动。

- 命令行（**xcrun simctl** 命令）：

  - xcrun simctl list —— 列出所有可用的模拟器。

  - xcrun simctl boot "iPhone 15" —— 启动指定的模拟器。

  - xcrun simctl shutdown all —— 关闭所有正在运行的模拟器。

> 频繁地启动和管理模拟器是开发过程中很常见的事情，因此市面上也有很多应用可以帮助你在不打开 Android Studio 和 Xcode 的情况下进行操作。
>
> 我们日常使用的一些应用包括：
>
> - MiniSim
>
> - Android iOS Emulator for VSCode
>
> - Expo Orbit
>
> - Shopify Tophat
