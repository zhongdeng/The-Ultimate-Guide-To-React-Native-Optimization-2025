# 关闭 JavaScript Bundle 压缩

我们之前已经了解到，APK 和 Android App Bundles（AAB）本质上是压缩归档文件。Android 构建系统在打包过程中会对文件进行压缩，主要目的是减少 APK 或 AAB 的整体体积，从而降低下载量并节省设备上的存储空间。这个压缩过程通常不会特别区分文件格式，除了跳过那些已经高效压缩的格式，比如图像文件（例如 .png、.jpg）。

在一个使用 Hermes 的 Android React Native 应用中，最终的 JS bundle 被打包进 index.android.bundle 文件中。虽然这是一个二进制文件，但它可能会被构建系统像其他资源一样处理，因为系统默认会压缩大多数资源文件，除非被明确告知不要这么做。

令人关注的是，由于该文件被压缩，它会在内存映射（mmap）过程中被跳过。这个问题在一个提交到 React Native 核心仓库的 [Pull Request](https://github.com/facebook/react-native/pull/49449) 中被详细调查。这种行为会阻止 Hermes 的一个关键优化机制生效，从而对应用的 TTI（Time to Interactive）指标产生负面影响。

> 截至撰写本文时（2025 年 2 月 17 日），React Native 核心团队已将此行为默认包含在内，并极有可能在 React Native 0.79 版本中提供。

无论你使用的是哪个版本的 React Native，都可以选择手动关闭这种默认行为，从而充分发挥 Hermes 的潜力。你可以在 app/build.gradle 文件中配置 androidResources，让 Android 构建系统不要压缩以 .bundle 为扩展名的文件：

```diff
android {
+ androidResources {
+   noCompress += ["bundle"]
+ }
}
```

这样一来，index.android.bundle 文件将不会被压缩，从而会增加整体应用包的体积。这个体积增加会影响应用的安装大小，但不会影响下载大小。最关键的是，由于 Hermes 能够启用内存映射并更快地加载 JS bundle，你的应用将加载得更快。在我们的初步测试中，一个安装体积为 75.9 MB 的 APK 文件，在禁用压缩后体积增加了 6.1 MB（+8%），而 TTI 降低了 450 毫秒（-16%）。

尽管存在一定可能性，但并不太常见，你的应用体积在禁用压缩后不会显著增长，这取决于多种因素。建议你实际测量结果，并观察用户的行为表现，以确保这个改动带来的利大于弊。但从整体情况来看，这种优化的收益很可能会大于其代价。
