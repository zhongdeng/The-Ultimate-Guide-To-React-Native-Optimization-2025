# 按需远程加载代码

在 Web 端，开发者天然就习惯于从远程服务器加载代码。毕竟，用户就是通过客户端（比如浏览器）从服务器加载 HTML、CSS、JS 和其他资源来访问应用的。但在移动端，情况稍有不同，这种模式更难以执行。用户通常是通过应用商店下载应用，JavaScript 代码已经包含在应用中并且是离线可用的。然而，在 React Native 中，我们仍然可以利用远程代码加载。只是，在使用它时，你需要采用不同于 Web 的思维方式。

## 何时考虑远程代码加载

从远程动态加载代码（即代码拆分）的效果，在很大程度上取决于你选择的 JavaScript 引擎。如果你使用的是 Hermes（React Native 的默认引擎），由于其内存映射技术已经可以高效地按需读取字节码 bundle 中必要的部分，因此按需加载代码带来的好处是极其有限的。但如果你使用的是 JavaScriptCore 或 V8，引入代码拆分可以显著提升初始加载时间表现，因为它允许按需加载代码块。

因此，在现代 React Native 应用中，大多数情况下代码拆分带来的收益并不大——但仍然存在一些适合的场景。你可以在以下情况下考虑尝试代码拆分：

- 你没有使用 Hermes 引擎。
- 你有特定的远程代码加载需求。
- 应用体积对用户体验产生了明显影响。
- 应用体积超过了 Google Play 商店 200MB 的限制。
- 你正在构建基于模块联邦（Module Federation）的微前端架构。
- 你已经用尽了其他优化手段

## 迁移到 Re.Pack

默认的 Metro bundler 并不原生支持生产环境下的代码拆分。如果你想实现这一特性，需要切换到 [Re.Pack](https://re-pack.dev/)——这是一个基于 Webpack 构建的打包工具，专为 React Native 打包而设计。虽然迁移现有配置可能需要一定的额外工作，但如果你已经尝试过其他所有优化手段，它可能是值得的。

要开始使用 Re.Pack，最好的方法是使用它的初始化工具：

```bash
npx @callstack/repack-init
```

该命令会对你的项目文件进行必要的更改，并引导你完成配置流程。完成后，下次你运行 npm start，你的 JavaScript 将不再由 Metro，而是由 Re.Pack 的开发服务器提供，并使用 Webpack 或 Rspack 作为打包器。虽然 Re.Pack 尽量成为 Metro 的“即插即用”替代方案，但你可能仍会遇到一些不兼容问题，请务必参考官方[迁移指南](https://re-pack.dev/docs/getting-started/quick-start)。

> Re.Pack 是 Callstack 团队开发的工具。我们最近还为它加入了对 Rspack 的支持，大幅提升了构建速度，在大多数情况下甚至比 Metro 更快。

## 拆分 bundle

最常见的代码拆分方式是结合 React 的 lazy 函数和 Suspense 组件，使用动态导入语法。添加拆分点通常如下所示：

```js
// Before - regular import
import FeatureComponent from './FeatureComponent';

// After - creating a split point
const FeatureComponent = React.lazy(() =>
);i
```

这些更改会使 FeatureComponent 在打包时被拆分到一个单独的文件中，命名为 feature.chunk.bundle。

以下是将某个页面组件进行代码拆分的实际示例：

```diff
-import React from 'react';
+import React, { Suspense } from 'react';
-import SettingsScreen from './screens/SettingsScreen';

+const SettingsScreen = React.lazy(() =>
+ import(/* webpackChunkName: "settings" */ './screens/SettingsScreen')
+);

const App = () => {
  return (
+   <Suspense fallback={<LoadingSpinner />}>
      </Suspense>
+   <SettingsScreen />
  );
};
```

## 从远程位置懒加载代码

注释中的 webpackChunkName 用于给 chunk 命名，这在最后一步中非常有用——也就是告诉 Re.Pack 如何加载这个 chunk，它不再属于主 bundle，而是我们可以从远程服务器下载的资源：

```js
// index.js
import { ScriptManager, Script } from "@callstack/repack/client";

// Tell Re.Pack where to load chunks from
ScriptManager.shared.addResolver((scriptId) => ({
  url: __DEV__
    ? // In development, load from the dev server
      Script.getDevServerURL(scriptId)
    : // In production, load from a CDN
      `https://my-cdn.com/assets/${scriptId}`,
}));

// Then, run the app just like before
AppRegistry.registerComponent(appName, () => App);
```

**ScriptManager** 是 Re.Pack 的运行时客户端，你可以用它来配置主 bundle 之外的其他 chunk 的查找和加载行为。这些步骤已经覆盖了代码拆分最基本的使用场景。

> 在实践中，使用代码拆分还涉及许多其他方面，比如缓存策略或部署到生产 CDN。请参考 [Re.Pack](https://re-pack.dev/) 官方文档获取更详细的信息，以实现你期望的代码拆分效果。

## 模块联合（Module Federation）

代码拆分的一个更具吸引力的用例是根据用户需求按需加载特性，而不对不需要的用户加载它们。例如，在大型企业级应用中，不同的用户角色可能需要访问完全不同的功能集合。与其将所有功能打包在一起，你可以根据用户权限或偏好按需加载模块。这不仅优化了初始加载速度，还能提供更个性化的体验。

对于更加复杂的场景，比如你需要管理多个彼此独立的团队，每个团队负责不同模块的开发与部署，你可以考虑使用模块联邦（Module Federation）构建微前端架构。这是目前 Web 和移动端中广泛验证的方案，得益于 Re.Pack，它在移动端也成为可能。

![](../assets/98.png "Monolith architecture vs decentralised one with Module Federation")

不过要注意，不要仅为了优化 bundle 体积而选择微前端架构。这种架构本身带来了额外的复杂性，只有在组织结构和开发流程的实际收益大于这类复杂性的前提下，才值得使用。

> 微前端的一个典型难点是版本管理。根据我们的经验，这一点如果自己手动实现会非常繁琐。如果你愿意使用第三方服务，我们推荐 [Zephyr Cloud](https://zephyr-cloud.io/)——它简化了这个问题，支持亚秒级部署，并与 Re.Pack 实现了官方集成。
