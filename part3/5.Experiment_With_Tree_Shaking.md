# 尝试使用 Tree Shaking 精简代码

Tree shaking 是现代 JavaScript 打包器中用于消除无用代码（dead code）的一种技术。这个术语来源于“摇树掉叶子”，就像从模块中“摇掉”那些未被使用的代码一样。

## Tree shaking 的工作原理

Tree shaking 通过以下两个步骤识别并移除你应用中未使用的代码：

1. 查找无用代码

   - 查找被导出但从未被导入的代码
   - 检查移除这些代码是否安全
   - 创建代码依赖关系映射图

2. 移除无用代码
   - 将未使用的代码标记为待删除
   - 在构建优化阶段将其移除
   - 最终只保留应用实际使用的代码

来看下面这个示例代码：

```js
// math.js
export const add = (a, b) => a + b;
export const subtract = (a, b) => a - b;

// app.js
import { add } from "./math";
```

在经过 tree shaking 处理后，subtract 会被移除，因为应用中并未使用它。虽然这是个简单示例，但像 Webpack、Rspack、ESBuild 等打包器已经能够很好地处理更复杂的函数和模块依赖关系。

> 如果一个库是以 EcmaScript Module（ESM）形式构建的，tree shaking 的效果会更好。而在历史上，React Native 一直使用 CommonJS 模块系统，并通过其 Babel 预设默认将 ESM 库转换为 CommonJS 格式。React Native 生态中的许多库依赖这种转换，缺少它可能会导致运行异常。

在生产模式下，打包器会执行以下四种主要优化：

1. **Used Exports Check** — 查找哪些导出实际被使用，移除未使用的。
2. **Side Effects Check** — 识别修改外部状态的代码，确保安全移除无用代码。
3. **Export Tracking** — 追踪模块之间的连接及 re-export 的情况。
4. **Variable Usage** — 跟踪变量的使用方式，防止误删有用代码。

## React Native 生态中的 Tree Shaking

在不同的 React Native 开发环境中，Tree shaking 的支持程度如下：

- **Metro** — 当前不支持 Tree shaking，但可以借助 rnx-kit 的 metro-serializer-esbuild 模拟实现。
- **Expo** — 从 Expo SDK 52 起，支持 Tree shaking 的实验性功能。
- **Re.Pack** — 通过集成 Webpack 和 Rspack，完全支持 Tree shaking。

> React Native 天生是一个多平台框架。借助 **Platform** 模块，可以定义特定平台的行为。Expo 和 Metro 正是利用这一特性实现了所谓的 [platform-shaking](https://docs.expo.dev/guides/tree-shaking/#platform-shaking) —— 移除特定平台代码（如 **iOS**、**Android** 或 **Web**）相关的模块。要实现这一功能，必须从 **react-native** 包中直接导入 **Platform** 模块。

### Expo

要在 Expo SDK 52 中启用 tree shaking，请在你的 Metro 配置中启用 experimentalImportSupport，并确保你的应用能够正常构建和运行。你可能会遇到循环依赖或混用 CommonJS 与 ESM 导入方式的问题。

```js
const { getDefaultConfig } = require("expo/metro-config");

const config = getDefaultConfig(__dirname);

config.transformer.getTransformOptions = async () => ({
  transform: {
    experimentalImportSupport: true,
  },
});

module.exports = config;
```

实验性的导入支持使用了 Metro 版本的 **@babel/plugin-transform-modules-commonjs** 插件，大幅减少了依赖解析的数量，并简化了输出的 bundle。这个功能可以与 inlineRequires 一起使用（在 Expo 项目中默认是关闭的），以进一步尝试性地优化你的 bundle。

然后，编辑你的 .env 文件，启用 `EXPO_UNSTABLE_METRO_OPTIMIZE_GRAPH` 和 `EXPO_UNSTABLE_TREE_SHAKING` 变量：

```.env
EXPO_UNSTABLE_METRO_OPTIMIZE_GRAPH=1
EXPO_UNSTABLE_TREE_SHAKING=1
```

这些设置仅在生产模式下生效。

### Re.Pack

Re.Pack 是目前 React Native 生态中最成熟的 Tree shaking 实现，得益于 Webpack 长期以来对该功能的支持。它默认启用 Tree shaking，并且与 Rspack 兼容，无需额外配置。

## 我们能预期什么样的改进？

我们使用 [Expensify](https://github.com/Expensify/App)（一款大型开源 React Native 应用）建立了全面的基准测试，比较了 Metro 与 Re.Pack 打包器下的 bundle 大小。测试包含了以下几种打包变体：

- **Development** — 含开发者工具的调试版本
- **Production** — 优化后的发布版本
- **Production Minified** — 使用 Terser 压缩的发布版本；
- **Production (HBC)** — 发布版本转为 Hermes 字节码；
- **Production Minified (HBC)** — 压缩后再转为 Hermes 字节码。

> 我们在相同代码库上测试了 Expo 的 Tree shaking 功能，虽然很想将结果纳入对比，但结果不够明确。在与 Metro 的对比中，我们发现 bundle 体积出现了回退，推测是配置不当导致的。由于该功能仍属实验性质，我们最终没有将其结果纳入对比。

Re.Pack 的生产包体积略大（+8.00%）可能令人意外，但这是预期中的结果，因为 Re.Pack（借助 Rspack）只标记未使用的代码，最终在压缩阶段才删除。而将 bundle 转换为 Hermes 字节码的过程包括了压缩，因此我们看到压缩包减少了 11% 到 14%，即使未提前使用 Terser 压缩。

总体而言，在启用所有优化的情况下，使用 Re.Pack 可将最终 bundle 体积减少约 **10%** 到 **15%**。虽然我们没有测量 TTI（首次交互时间）的变化，但根据 Hermes 字节码大小的减少，我们可以预期启动时间略微加快，内存占用也更低。
