# 实现不掉帧的高性能动画

动画在移动体验中起着至关重要的作用。在 React Native 中，确保动画平稳运行、至少保持 60 帧每秒（FPS）是非常重要的，但在处理高负载操作时有时会面临挑战。当动画掉帧或表现卡顿时，会显著影响用户体验，让你的应用显得不够精致。在本章中，我们将介绍一些用于确保动画按预期运行的工具，它们的局限性，以及应遵循的最佳实践。

## 理解 React Native 中的主线程

首先，让我们了解什么是主线程。在 React Native 中，主线程（也称为 UI 线程）是负责处理 UI 渲染和用户交互的主要线程。所有的视觉更新都发生在这个线程上，用户界面对触摸事件的响应也在这里进行。

> 在 React Native 中，“主线程”和“UI 线程”这两个术语是可以互换使用的。开发者在提到其中任何一个时，指的都是同一件事。

这与 JavaScript 线程不同，大多数 React Native 代码运行在 JavaScript 线程上，我们在本书的介绍章节中已经做过更广泛的描述。

## React Native Reanimated

当你想为 React Native 应用添加动画时，应选择 React Native Reanimated，这是一个功能强大的库，提供对高性能动画的一流支持。

> [React Native Reanimated](https://docs.swmansion.com/react-native-reanimated/) 已成为创建动画的业界标准。截至 2025 年 1 月，它是一个经过大量实战验证的解决方案，每周下载量接近 100 万次。

借助 Reanimated，你可以轻松地在 UI 线程上运行动画，这得益于其 “worklets” 概念。Worklet 是可以在 UI 线程上运行的短小 JavaScript 函数，也可以像普通函数一样在 JavaScript 线程上运行。

> Worklet 概念由 Reanimated 首创，后来被其他社区库采用，例如 VisionCamera 或 LiveMarkdown。这些库可以使用 **createWorkletRuntime** API 来创建新的 JS 运行时，从而在非 JS 或 UI 线程上运行 worklet。

大多数情况下，当你使用 Reanimated 时，代码默认是在 UI 线程上运行的，例如使用 `useAnimatedStyle` hook 时就是如此：

```js
const style = useAnimatedStyle(() => {
  console.log("Running on the UI thread");
  return { opacity: 0.2 };
});
```

Reanimated 提供了各种辅助函数，其中我们特别想强调的是 **runOnUI**。它允许你手动在 UI 线程上运行任意代码。例如可以与 useEffect 搭配使用，在组件挂载/卸载时启动动画，或者与 **measure** 和 **scrollTo** 函数一起使用，这些函数仅在 UI 线程中实现。你可以这样使用它：

```js
runOnUI(() => {
  console.log("Running on the UI thread");
});
```

还有另一个名为 runOnJS 的实用函数，它可以用来运行不是 worklet 且不能在 UI 线程上运行的函数，例如来自外部库的函数。

```js
// Function that needs to run on JS thread
const notifyCompletion = () => {
  console.log("Animation completed!");
  // Could be analytics tracking, state updates, etc.
};

const animatedStyle = useAnimatedStyle(() => {
  if (progress.value >= 1) {
    // Call JS thread function from UI thread
    runOnJS(notifyCompletion)();
  }
  return {
    opacity: progress.value,
    transform: [{ scale: withSpring(progress.value) }],
  };
});
```

应在哪个线程上运行代码的选择取决于多个因素：

**UI 线程（主线程）**

- 适用于视觉动画、变换和布局变更。

- 当你需要直接操作 UI 元素时使用。

**JavaScript 线程**

- 适用于复杂计算、数据处理和状态管理。

- 当你需要访问 React 的状态或 props，或执行业务逻辑时使用。

## InteractionManager

React Native 自带了 **InteractionManager** API，它允许在所有交互/动画完成之后再调度执行耗时任务。这有助于让 JavaScript 动画更加流畅，避免卡顿和掉帧。

当用户与应用交互（例如点击按钮或操作手势）时，如果 JavaScript 线程同时执行繁重任务和 UI 更新，可能会引发性能问题和异常行为。使用 **InteractionManager**，我们可以将非关键任务延后到当前所有交互完成之后再执行：

```js
InteractionManager.runAfterInteractions(() => {
  console.log("Running after interactions");
});
```

触摸处理系统将一个或多个激活的触摸视为一次“交互”，并会推迟 `runAfterInteractions()` 的回调，直到所有触摸结束或被取消。

你还可以使用 `InteractionManager.createInteractionHandle` 方法注册动画，在交互完成后手动清除它，从而触发所有已调度的操作。

```js
const handle = InteractionManager.createInteractionHandle();
// run animation... (`runAfterInteractions` tasks are queued)
// later, on animation completion:
InteractionManager.clearInteractionHandle(handle);
// queued tasks run if all handles were cleared
```

### 与 React Navigation 搭配使用

一个典型的使用 **InteractionManager** 的场景是配合 React Navigation 的 `useFocusEffect` 使用。这个 hook 会在屏幕进入焦点时立即执行效果函数。这通常意味着，如果屏幕切换带有动画，该动画可能尚未完成。

React Navigation 的动画是在 UI 线程上运行的，因此很多情况下并不会造成问题。然而，如果效果函数会更新 UI 或执行开销较大的操作，可能会影响动画性能。在这种情况下，我们可以使用 **InteractionManager** 来延后这些操作，直到动画或手势完成：

```js
useFocusEffect(
  React.useCallback(() => {
    const task = InteractionManager.runAfterInteractions(() => {
      // Expensive task eventually updating UI
    });
    return () => task.cancel();
  }, [])
);
```

除了 React Native 提供的 **InteractionManager** API，你也可以使用 React 的最新特性，例如 `startTransition` hook，将非关键更新延后执行，以避免阻塞 UI 更新。你可以在[《使用 `过渡` 来处理非关键更新》](./7.Concurrent_React.md#使用-过渡-来处理非关键更新)一章中了解更多内容。
